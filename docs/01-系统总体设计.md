# 应用中心系统总体设计

## 1. 项目概述

### 1.1 项目背景
应用中心是一个企业级软件管理平台，旨在帮助管理员集中管理企业内部软件的分发和安全监控。系统采用B/S架构的管理后台和C/S架构的桌面客户端相结合的模式，实现软件的统一管理、分发和安全管控。

**技术选型特点**：
- **高性能**：使用Rust开发后端，获得接近C++的性能和内存安全性
- **轻量级客户端**：使用Tauri开发桌面应用，客户端体积仅5-10MB
- **跨平台**：一套代码支持Windows、macOS、Linux三大平台
- **现代化**：使用最新技术栈，代码质量高，维护性强

### 1.2 项目目标
- 实现软件安装包的集中存储和管理
- 提供便捷的软件分发和下载功能
- 监控客户端进程安全，防止违规软件运行
- 提供完善的用户权限管理体系
- 保证系统的安全性、可靠性和高性能

### 1.3 技术选型优势

| 维度 | Rust + Tauri方案 | 传统方案对比 |
|------|------------------|--------------|
| **后端性能** | 接近C++性能，内存占用低 | Java内存占用较高 |
| **客户端体积** | 5-10MB（Tauri） | 50-100MB（PyQt/Electron） |
| **内存安全** | Rust编译时保证内存安全 | Java运行时GC |
| **跨平台** | 原生支持Win/Mac/Linux | PyQt主要支持Windows |
| **部署简单** | 单一二进制文件 | 需要JVM/Python环境 |
| **并发性能** | async/await零成本抽象 | 线程池开销较大 |

### 2.2 技术架构

#### 2.2.1 Server端技术栈（Rust）
- **核心框架**: Axum 0.7+ / Actix-web 4.0+（高性能Web框架）
- **编程语言**: Rust 1.70+（内存安全、零成本抽象）
- **数据库**: PostgreSQL 15+ / MySQL 8.0+（企业级数据库）
- **ORM框架**: SeaORM 0.12+ / Diesel 2.0+（异步ORM）
- **认证授权**: JWT + 自定义中间件（无运行时开销）
- **异步运行时**: Tokio 1.0+（高性能异步运行时）
- **文件存储**: 本地文件系统 / MinIO（可扩展）
- **日志框架**: tracing / tracing-subscriber（结构化日志）
- **序列化**: serde + serde_json（零成本序列化）
- **配置管理**: config-rs（类型安全配置）
- **API文档**:utoipa（OpenAPI自动生成）
- **构建工具**: Cargo（包管理器 + 构建工具）

#### 2.2.2 Client端技术栈（Tauri）
- **开发框架**: Tauri 2.0+（Rust + WebView）
- **前端框架**: Vue 3 / React / Svelte（现代化UI）
- **状态管理**: Pinia / Redux / Svelte stores
- **HTTP客户端**: axios / fetch API
- **进程监控**: Rust sysinfo crate（跨平台进程监控）
- **系统交互**: Tauri Commands（IPC通信）
- **打包工具**: Tauri CLI（自动打包）
- **权限管理**: Tauri Capabilities（细粒度权限控制）

#### 2.2.3 为什么选择Rust + Tauri？

**Rust优势**：
1. **性能卓越**: 
   - 启动速度快（毫秒级 vs Java秒级）
   - 内存占用低（10-50MB vs Java 100-300MB）
   - CPU效率高（零成本抽象）
   
2. **内存安全**:
   - 编译时检查，无运行时GC
   - 防止空指针、内存泄漏、数据竞争
   - 适合长期运行的服务
   
3. **并发模型**:
   - async/await异步编程
   - 无锁数据结构
   - 高并发处理能力
   
4. **部署简单**:
   - 单一二进制文件
   - 无运行时依赖
   - 跨平台编译

**Tauri优势**：
1. **体积小**: 5-10MB（vs Electron 100MB+）
2. **性能好**: 使用系统WebView，内存占用低
3. **安全性高**: 多层沙箱隔离
4. **开发体验好**: 前端技术栈（Vue/React）
5. **跨平台**: Win/Mac/Linux一套代码

### 3.2 核心功能详细说明

#### 3.2.1 用户管理
**管理员功能：**
- 用户注册：管理员创建新用户账号
- 用户列表：查看所有用户信息（分页查询）
- 权限分配：基于RBAC模型的权限管理
- 状态管理：启用/禁用用户账号

**普通用户功能：**
- 个人信息：查看和修改个人信息
- 密码修改：修改登录密码（BCrypt加密）

**Rust实现特点**：
- 使用SeaORM实现用户CRUD
- 密码使用bcrypt crate加密
- 权限检查通过中间件实现（无运行时开销）

#### 3.2.2 软件管理
- **软件上传**：
  - 支持大文件分片上传（>500MB）
  - 使用tokio::fs异步文件操作
  - MD5校验使用ring crate
  - 支持断点续传
  
- **软件信息**：软件名称、版本、大小、描述、分类等
- **软件分类**：按类型分类管理（办公、开发、工具等）
- **版本控制**：保留历史版本，支持回滚

**Rust性能优势**：
- 文件上传使用异步流（Stream）
- 大文件处理内存占用恒定
- 并发上传处理能力强（零成本异步）

#### 3.2.3 黑名单管理
- **进程黑名单**：配置禁止运行的进程名称列表
- **黑名单分组**：支持按部门/用户组配置不同黑名单
- **实时推送**：黑名单更新后实时推送到客户端
- **版本控制**：使用版本号机制，客户端增量同步

**Rust实现特点**：
- 使用DashMap实现高并发黑名单缓存
- 无锁读取，极致性能
- 版本号使用AtomicU64原子操作

#### 3.2.4 监控与报表
- **客户端在线状态**：实时监控客户端连接状态
- **异常上报**：接收客户端黑名单进程检测报告
- **统计分析**：生成软件下载统计、异常统计等报表
- **日志追踪**：使用tracing记录所有操作

#### 3.2.5 客户端安全监控
- **定时扫描**：按配置间隔扫描系统进程（默认5分钟）
- **黑名单匹配**：对比运行进程与黑名单列表
- **异常上报**：发现异常进程立即上报服务器
- **本地日志**：使用tracing-appender记录日志
- **管理员权限**：以管理员身份运行，获取完整进程访问权限

**Tauri + Rust实现**：
- 使用sysinfo crate跨平台获取进程列表
- Tauri Command调用Rust扫描函数
- 前端通过IPC触发扫描
- 管理员权限配置在tauri.conf.json

## 5. 非功能性需求

### 5.1 性能要求
- 系统支持至少1000个客户端同时在线
- 软件下载速度：局域网内不低于50MB/s（Rust异步IO）
- API响应时间：95%的请求在100ms内响应（vs Java 500ms）
- 进程扫描对客户端性能影响小于1% CPU占用（Rust高效）

### 5.2 安全要求
- 所有通信使用HTTPS加密（TLS 1.3）
- 密码采用BCrypt加密存储（bcrypt crate）
- API接口使用JWT认证（jsonwebtoken）
- 敏感操作记录审计日志（tracing）
- 防止SQL注入（SeaORM编译时检查）
- XSS防护（前端框架内置）

### 5.3 可用性要求
- 系统可用性达到99.9%（Rust内存安全）
- 支持数据备份和恢复
- 客户端支持离线模式（仅本地功能）

### 5.4 可扩展性
- 支持水平扩展（无状态设计）
- 文件存储可切换到分布式存储
- 数据库支持读写分离（SeaORM支持）

## 6. 系统交互流程

### 6.1 管理员上传软件流程

```
管理员 -> 浏览器 -> Rust Server
1. 管理员登录系统（JWT认证）
2. 进入软件管理页面
3. 选择文件上传（前端发起请求）
4. 填写软件信息（名称、版本、描述等）
5. 提交上传
6. Server验证权限和文件（中间件 + 异步验证）
7. 分片接收文件（tokio::fs异步写入）
8. 合并文件并计算MD5（ring crate）
9. 保存文件到存储系统（异步IO）
10. 记录文件信息到数据库（SeaORM异步）
11. 返回上传结果（JSON序列化）
```

### 6.2 用户下载软件流程

```
用户 -> Tauri Client -> Rust Server
1. 用户启动客户端（Tauri应用）
2. 用户登录（账号认证，获取JWT）
3. 获取软件列表（API请求，前端渲染）
4. 选择需要下载的软件
5. 发起下载请求（axios/fetch）
6. Server验证下载权限（中间件）
7. 返回下载链接（生成临时URL）
8. Tauri接收并保存文件（Rust异步IO）
9. 显示下载进度（IPC通信）
10. 下载完成提示
```

### 6.3 进程监控上报流程

```
Tauri Client (Rust) -> Rust Server
1. 定时器触发扫描（前端定时器）
2. 调用Tauri Command（IPC）
3. Rust扫描进程（sysinfo crate）
4. 获取当前系统所有进程列表
5. 从本地缓存获取黑名单（DashMap）
6. 对比进程与黑名单
7. 生成上报数据（struct序列化）
8. 发送HTTP请求到Server（reqwest）
9. Server记录上报日志（SeaORM + tracing）
10. 如有异常，触发告警通知
```

### 6.4 管理员权限处理流程

```
Windows系统进程监控权限
1. Tauri客户端启动
2. 检查是否管理员权限
3. 如果不是管理员：
   - 弹出提示框
   - 提示用户"以管理员身份运行"
   - 可配置自动请求提权
4. 如果是管理员：
   - sysinfo获取所有进程（包括系统进程）
   - 执行黑名单匹配
   - 正常上报
```

- Rust: 1000进程扫描耗时10-20ms（sysinfo）

- Java: 1000并发请求处理耗时5-10秒（线程池）

## 12. 技术难点解决方案

### 12.1 管理员权限问题

**Windows进程监控权限解决方案**：

```json
// tauri.conf.json权限配置
{
  "tauri": {
    "allowlist": {
      "all": false,
      "process": {
        "all": true,
        "relaunch": true,
        "exit": true
      },
      "shell": {
        "all": false,
        "execute": true,
        "sidecar": true,
        "open": true
      }
    },
    "security": {
      "capabilities": [
        "process-monitor",
        "admin-mode"
      ]
    },
    "updater": {
      "active": true
    }
  }
}
```

**实现步骤**：
1. 检测是否管理员权限（Windows API）
2. 非管理员时弹出提示框
3. 用户可选择"以管理员身份重启"
4. 管理员权限下获取完整进程列表
5. 执行黑名单匹配和上报

### 12.2 Rust开发难点应对

**缺少成熟框架问题**：
- 使用Axum/Actix-web替代Spring Boot
- 使用SeaORM替代MyBatis-Plus
- 自定义中间件替代Spring Security
- 使用utoipa自动生成API文档

**学习曲线问题**：
- 提供详细代码示例和文档
- 使用成熟crate减少自定义代码
- 团队技术培训和外部顾问支持
- 渐进式开发（先核心功能，再扩展）

**调试困难问题**：
- 使用tracing结构化日志
- 使用调试器rust-gdb/rust-lldb
- 单元测试和集成测试完善
- 类型系统编译时检查减少运行时错误
