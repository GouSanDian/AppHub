# 数据库设计文档

## 1. 数据库概述

### 1.1 数据库选型

#### 主要数据库
- **数据库类型**: PostgreSQL 15+ (推荐) / MySQL 8.0+ (支持)
- **字符集**: UTF8 (PostgreSQL) / utf8mb4 (MySQL)
- **排序规则**: UNICODE (PostgreSQL) / utf8mb4_unicode_ci (MySQL)
- **存储引擎**: InnoDB (MySQL)

#### PostgreSQL推荐理由（Rust生态）
- **SeaORM原生支持**: SeaORM对PostgreSQL支持更好
- **性能优异**: PostgreSQL在高并发场景性能更好
- **功能丰富**: 支持JSONB、数组、全文搜索等高级特性
- **扩展性强**: 支持自定义类型、函数、索引
- **开源生态**: 与Rust开源生态契合度高

### 1.2 ORM框架选择

#### SeaORM (推荐)
- **特点**: 异步ORM，完全异步设计
- **性能**: 基于SQLx，性能接近原生SQL
- **类型安全**: 编译时检查SQL语法
- **迁移支持**: 内置迁移工具
- **适用场景**: 高性能异步应用

#### Diesel (可选)
- **特点**: 编译时检查SQL，极致类型安全
- **性能**: 性能最优，零运行时开销
- **缺点**: 异步支持较弱，迁移工具复杂
- **适用场景**: 对性能要求极致的场景

### 1.3 命名规范
- 表名使用小写字母，单词间用下划线分隔（snake_case）
- 主键统一命名为 `id`（SeaORM默认约定）
- 外键命名格式：`表名_id`
- 时间字段统一使用 `created_at`, `updated_at`（Rust惯例）
- 逻辑删除字段统一使用 `deleted` (0-未删除, 1-已删除)
- Rust模型使用CamelCase命名，数据库表使用snake_case

## 2. 数据表设计

### 2.1 用户表 (sys_user)

用于存储用户账号信息。

| 字段名 | 类型 | 长度 | 允许空 | 默认值 | 说明 |
|--------|------|------|--------|--------|------|
| id | BIGINT | - | 否 | 自增 | 主键ID |
| username | VARCHAR | 50 | 否 | - | 用户名（唯一） |
| password | VARCHAR | 255 | 否 | - | 密码（BCrypt加密） |
| nickname | VARCHAR | 100 | 是 | NULL | 昵称 |
| email | VARCHAR | 100 | 是 | NULL | 邮箱 |
| phone | VARCHAR | 20 | 是 | NULL | 手机号 |
| avatar | VARCHAR | 255 | 是 | NULL | 头像URL |
| status | TINYINT | - | 否 | 1 | 状态（0-禁用, 1-启用） |
| role_id | BIGINT | - | 否 | - | 角色ID |
| department_id | BIGINT | - | 是 | NULL | 部门ID |
| last_login_time | DATETIME | - | 是 | NULL | 最后登录时间 |
| last_login_ip | VARCHAR | 50 | 是 | NULL | 最后登录IP |
| create_time | DATETIME | - | 否 | CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | - | 否 | CURRENT_TIMESTAMP | 更新时间 |
| deleted | TINYINT | - | 否 | 0 | 逻辑删除（0-未删除, 1-已删除） |

**索引设计：**
- PRIMARY KEY (id)
- UNIQUE KEY uk_username (username)
- KEY idx_role_id (role_id)
- KEY idx_department_id (department_id)
- KEY idx_create_time (create_time)

## 6. 数据库优化建议

### 6.1 索引优化
- 为高频查询字段建立索引（SeaORM自动创建）
- 组合索引遵循最左匹配原则
- 避免在低选择性字段建立索引
- PostgreSQL支持部分索引（Partial Index）和表达式索引

### 6.2 查询优化（Rust + SeaORM）
- 使用SeaORM的异步查询（async query）
- 避免SELECT *，只查询需要的字段
- 合理使用分页查询（.paginate()）
- 复杂查询考虑使用缓存（DashMap）
- 使用SQL原生查询处理复杂场景（sea_orm::Statement）

### 6.3 存储优化
- 大文本字段（TEXT）考虑单独存储
- PostgreSQL支持JSONB字段，减少关联查询
- 历史数据定期归档（pg_partman扩展）
- 日志表按时间分区（Partitioning）

### 6.4 连接池优化（SeaORM）
- 配置合适的连接池大小（SqlxPoolOptions）
- 设置最大连接数和最小连接数
- 配置连接超时和空闲超时
- 使用PgBouncer连接池中间件（可选）

### 6.5 SeaORM性能优化示例

let page_data = paginator.fetch_page(1).await?;

    },

    .exec(db)
    .await?;

## 7. 安全优化

### 7.1 敏感字段加密

- API密钥使用加密存储（可选）
- Token使用JWT签名（jsonwebtoken crate）

### 7.2 数据库安全
- 数据库账号权限最小化（只授予必要权限）
- 定期备份数据（pg_dump / mysqldump）
- 使用SSL连接数据库（加密传输）
- 防止SQL注入（SeaORM编译时检查）

### 7.3 SeaORM安全优势
- **编译时检查**: SeaORM在编译时检查SQL语法，防止SQL注入
- **类型安全**: Rust类型系统防止数据类型错误
- **参数化查询**: 自动使用参数化查询，避免拼接SQL
- **事务安全**: 提供事务API，保证数据一致性
